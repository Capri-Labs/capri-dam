=== Workflow Automation & Approvals

==== Overview

A visual, drag-and-drop workflow designer lets non-developers build asset approval
and automation pipelines: who reviews what, under which trigger conditions, and
what happens next.

==== Key features

[cols="2,3", options="header"]
|===
| Feature | Detail

| Visual workflow designer | DAG-based canvas (`@xyflow/react`) for building approval and automation flows, with a searchable node toolbox organised into categories.
| Step / node types | Approval (human task), Parallel Review, Sequential Review, Send Email, In-App Alert, Slack Message, Delay, Webhook, Condition (TRUE/FALSE branching), Start/End.
| Trigger rules | Metadata-based predicates — e.g. On Asset Upload — with scope control (all folders, or specific folders with exclusions).
| Blueprint validation | Inline validation before publish (e.g. an Approval step must have an assignee).
| Task assignment | Assign tasks to users or groups; email + in-app notification, due dates.
| Workflow dashboard | Separate dashboard view of active/completed workflow tasks (`/workflows/dashboard`).
| Workflow-protected purge | Assets with active workflow instances can't be purged from the Recycle Bin (see <<module-recycle-bin>>).
|===

==== Powerful / notable capabilities

[.feature-highlight]
=====

* **No-code DAG designer** — branching logic (Condition nodes with TRUE/FALSE
  outputs), parallel/sequential review, and multi-channel notifications are all
  assembled visually, with zero code.
* **Inline publish-time validation** — the designer actively prevents you from
  publishing a broken blueprint (e.g. an approval step nobody is assigned to)
  rather than failing silently at runtime.
* **Cross-cutting safety** — workflow state integrates with the Recycle Bin's purge
  guard, so in-review assets can't be accidentally destroyed.
=====


==== Notes & known limitations

* The canvas currently supports a fixed catalogue of node types; there is no
  plugin/custom-node SDK for teams wanting bespoke step types.
* Condition-node branching is binary (TRUE/FALSE) only — there's no multi-way
  switch/case branching yet.

==== Test coverage status

[.status-covered]#Covered by E2E# (`spec/e2e/workflow_designer.e2e.spec.js`)

* All 6 toolbox categories are shown; palette search filters nodes correctly.
* Dropping an Email node onto the canvas; dropping a Condition node and confirming
  TRUE/FALSE branch labels appear.
* Configuring a Delay node and a Webhook node's URL field.
* Filling in a blueprint name and successfully publishing.
* Validation error shown when an Approval step has no assignee.
* Regression checks confirming a removed legacy block ("System-Wide Escalation")
  no longer appears, and redesigned section headers render.
