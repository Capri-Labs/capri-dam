[[module-asset-management]]
=== Asset Management (Folders & Digital Assets)

==== Overview

The core of Capri DAM: an N-ary folder hierarchy holding versioned digital assets,
each with rich technical and custom metadata, access-controlled per folder.

image::folders-browser.png[Digital Assets browser, title="The 'All Assets' browser — grid view with folder cards, search, and view-density controls"]

image::folder-access-acl.png[Folder Access tab, title="Folder Info Panel — Access tab with per-group ACL, cascade toggle, and explicit-deny checkbox"]

==== Key features

[cols="2,3", options="header"]
|===
| Feature | Detail

| Upload & ingest | Drag-and-drop browser upload, direct-to-storage presigned PUTs, and bulk ingestion from 14 external sources (see <<module-ingestion>>).
| Folder hierarchy | N-ary tree with materialised paths; per-folder RBAC policies, metadata schemas, and image/video profiles.
| Grid & list views | Small / medium / large card density, plus a dedicated list/table view; sortable by name, date, size.
| Folder info panel | Drawer showing Properties, Metadata schema, Video Profiles, and Access (see <<module-access-control>>) tabs per folder.
| Version history | Immutable version snapshot created on every asset replacement, with a diff view in the UI.
| Metadata | Technical (EXIF/IPTC/XMP) auto-extracted on ingest; custom schema-driven fields configurable per folder; bulk CSV import/export.
| Move | Select any mix of folders/assets and relocate them to a new destination in one request (`POST /api/v1/move_operations`); per-item success/failure reporting; cycle-prevention guards against moving a folder into its own descendant.
| Copy | "Copy to…" action mirroring Move — duplicates the selection into a destination folder instead of relocating it (`POST /api/v1/copy_operations`). Folders are copied recursively (subfolders + assets); asset files share the source's existing ActiveStorage blob (no re-upload); destination name collisions are auto-resolved (`" (Copy)"`, `" (Copy 2)"`, …).
| Bulk/folder download | "Download" action on a folder or a multi-asset selection streams a ZIP built asynchronously by a background worker, with a live progress bar in the dialog. A second download requested while one is already running is queued rather than run concurrently; the finished ZIP link is delivered to the user's Inbox and notification bell so they can navigate there and download whenever it's ready.
| Publish / Unpublish | "Manage Publish" — a dedicated button/menu next to (not inside) *Tools*, shown only for an all-assets selection. Publish/Unpublish apply immediately (`POST /api/v1/assets/:id/publish`\|`/unpublish`); Publish Later/Unpublish Later instead schedule the action for a future date/time, applied automatically by a polling background worker.
| Quarantine | Security anomaly detection flags suspicious assets, queues them for admin review, then supports release back into the library or soft-delete into the Recycle Bin.
| Folder-scoped search & filters | Filter by content type and status, search by name, inside any folder scope.
|===

==== Powerful / notable capabilities

[.feature-highlight]
=====

* **Per-folder everything** — access policy, metadata schema, and video encoding
  profile are all scoped to a folder path and automatically inherited by its
  subfolders, so a single configuration change cascades correctly.
* **Immutable version history** — replacing an asset never destroys the previous
  file; every version remains inspectable and restorable.
* **Schema-driven custom metadata** — folders can define their own structured
  metadata fields (not just free-text tags), enabling consistent, queryable
  attributes across a whole content library.
* **Copy without re-upload** — bulk copy duplicates asset *records* and shares
  the existing file bytes via the underlying ActiveStorage blob, so copying a
  large folder tree doesn't re-transfer any file data.
* **Fire-and-forget bulk download** — the Download dialog can be closed the
  moment it's opened without losing the job; it's a background worker with
  its own progress state, so users are free to keep browsing while a large
  ZIP builds, then pick up the finished link later from their Inbox.
=====


==== Notes & known limitations

* Bulk metadata CSV import now supports a dry-run / preview-before-commit mode,
  returning per-row success/failure details before users launch the real import job.
* The version-history compare view now supports a pixel-level overlay for any two
  saved versions, with an automatic side-by-side fallback when browser CORS
  rules block canvas diffing for CDN-hosted previews.

==== Test coverage status

[.status-covered]#Covered by E2E#

* Folder Info Panel — Access tab (rendering, add/cancel group form, cascade toggle,
  explicit-deny checkbox, inherited-policy badges) — `spec/e2e/folder_access_tab.e2e.spec.js`.
* Local asset serving (`GET /api/v1/assets/local/:uuid`) — ETag/Cache-Control, 404s,
  304 Not-Modified, `?variant=` query param — `spec/e2e/asset_serving.e2e.spec.js`.
* Asset deep-link (`/assets?id=UUID`) rendering without a JS crash — `spec/e2e/asset_serving.e2e.spec.js`.
* Quarantine review workspace (queue rendering, release flow, discard-to-bin flow)
  — `spec/e2e/quarantine.e2e.spec.js`.
* Move — Tools menu overlay: moving a single folder into another folder, moving a
  single asset to root, and the Move option correctly disabling when the selection
  includes an item the user lacks delete rights on — `spec/e2e/move_folder_asset.e2e.spec.js`.
* Copy — Tools menu overlay: copying a single folder (original left in place),
  copying a single asset to root (original left in its folder), and the Copy option
  staying enabled even without delete rights (copy only needs read+create) —
  `spec/e2e/copy_folder_asset.e2e.spec.js`.
* Bulk/folder download — Tools menu overlay: downloading a single asset with progress
  through to a ready ZIP link, downloading a folder recursively, and the Download
  option staying enabled without delete rights — `spec/e2e/asset_download.e2e.spec.js`.
* Manage Publish — immediate publish then unpublish of a selected asset, scheduling a
  future "Publish Later" action without prematurely flipping the asset's published
  state, and the button correctly hiding whenever the selection includes a folder —
  `spec/e2e/publish_asset.e2e.spec.js`.
* Version history — listing versions and restoring a previous one via the real
  restore endpoint (`spec/e2e/folders_workspace_coverage.e2e.spec.js`), plus a
  real end-to-end pixel-diff run: uploading an asset, generating a second
  version via `process_image` (rotate, `save_mode: version`), and asserting the
  diff overlay renders real before/after canvases —
  `spec/e2e/asset_version_diff.e2e.spec.js`. Writing this real (non-mocked)
  test surfaced two genuine bugs, both now fixed:
  ** `AssetVersionsTab.jsx`'s `loadImage()` unconditionally set
     `image.crossOrigin = 'anonymous'` even for same-origin preview requests,
     which strips session cookies and breaks authenticated image loads for any
     self-hosted/dev deployment without a CDN in front of asset serving.
  ** `AssetUrlHelper#asset_url_for` bypassed the UUID-based local-storage
     routing guard whenever an explicit `version:` was requested, so it
     returned the raw absolute filesystem path instead of the
     `/api/v1/assets/local/:uuid?version_id=...` route — meaning **every**
     version-scoped preview URL was fundamentally broken (guaranteed 404) on
     any deployment using local storage, not just a display glitch.
* Bulk metadata CSV import/export — real end-to-end CSV upload/processing,
  results download, and CSV export/download —
  `spec/e2e/metadata_tools.e2e.spec.js`.
