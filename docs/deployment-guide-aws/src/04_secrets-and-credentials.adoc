== Secrets & Credentials

=== `RAILS_MASTER_KEY`

Capri DAM decrypts `config/credentials.yml.enc` using `RAILS_MASTER_KEY` at boot — this is the one
secret that must never be baked into the image or committed. Store it in **AWS Secrets Manager**:

[source,bash]
----
aws secretsmanager create-secret --name capri-dam/rails-master-key \
  --secret-string "$(cat config/master.key)"
----

Reference it from the ECS task definition's `secrets` block (not `environment`) so it's injected at
container start without ever appearing in `describe-task-definition` output in plaintext, CloudTrail,
or the console:

[source,json]
----
{
  "containerDefinitions": [{
    "name": "web",
    "secrets": [
      { "name": "RAILS_MASTER_KEY", "valueFrom": "arn:aws:secretsmanager:...:secret:capri-dam/rails-master-key" },
      { "name": "DATABASE_URL", "valueFrom": "arn:aws:secretsmanager:...:secret:capri-dam/database-url" },
      { "name": "REDIS_URL", "valueFrom": "arn:aws:secretsmanager:...:secret:capri-dam/redis-url" }
    ]
  }]
}
----

=== Other runtime configuration

[cols="1,1,3", options="header"]
|===
| Variable | Where | Notes
| `RAILS_MASTER_KEY` | Secrets Manager (`secrets`) | See above.
| `DATABASE_URL` | Secrets Manager (`secrets`) | Points at the Aurora writer endpoint — see <<section-aws-managed-data-services>>.
| `REDIS_URL` | Secrets Manager (`secrets`) | Points at the ElastiCache primary endpoint.
| `COVERBAND_REDIS_URL` | Secrets Manager (`secrets`) | Same ElastiCache endpoint, distinct logical DB index if desired.
| `RAILS_ENV` | Task definition `environment` (plain) | `production` — not sensitive, fine as plain env.
| `RAILS_LOG_TO_STDOUT` / `RAILS_SERVE_STATIC_FILES` | Task definition `environment` (plain) | `true` — CloudWatch Logs captures stdout; Thruster serves static assets.
|===

=== IAM roles

Two distinct roles are required per ECS task (this is a common point of confusion):

[cols="1,3", options="header"]
|===
| Role | Purpose
| **Task execution role** (`ecsTaskExecutionRole`) | Used by the ECS agent itself to pull the image from ECR and resolve the `secrets` block from Secrets Manager. Needs `AmazonECSTaskExecutionRolePolicy` plus `secretsmanager:GetSecretValue` scoped to the `capri-dam/*` secret ARNs.
| **Task role** (`capri-dam-task-role`) | Assumed by the application code itself inside the container — needed only if/when the app calls AWS APIs directly (e.g. S3 via the `aws` `StorageBackend` adapter). Scope it to `s3:GetObject`/`s3:PutObject`/`s3:DeleteObject` on the specific bucket from <<section-aws-managed-data-services>>, nothing broader.
|===

Never reuse a single overly-broad role for both purposes — keeping them separate limits the blast
radius if the container is ever compromised (the app process only ever gets S3 access to its own
bucket, never ECR/Secrets Manager access it doesn't need at runtime).

=== CI/CD IAM user (optional)

If your CI pipeline builds and deploys (see <<section-aws-routine-deployments-and-rollback>>), create
a dedicated IAM principal scoped to exactly: `ecr:GetAuthorizationToken`, `ecr:BatchCheckLayerAvailability`,
`ecr:PutImage`, `ecr:InitiateLayerUpload`/`UploadLayerPart`/`CompleteLayerUpload`,
`ecs:RegisterTaskDefinition`, `ecs:UpdateService`, and `iam:PassRole` limited to the two roles above —
never grant it account-wide `AdministratorAccess`.
