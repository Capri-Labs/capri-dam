[[section-aws-prerequisites]]
== Prerequisites

=== AWS account & access

[cols="1,3", options="header"]
|===
| Requirement | Recommendation
| AWS account | A dedicated account (or a dedicated OU/account under AWS Organizations) for this workload — avoids blast-radius overlap with unrelated projects.
| IAM user/role | An IAM principal with permissions to manage VPC, ECS, ECR, ELB, RDS/Aurora, ElastiCache, S3, Secrets Manager, and IAM roles (for CI, scope this down to exactly what `deploy` needs — see <<section-aws-secrets-and-credentials>>).
| Region | Pick one region for all resources in this guide (examples use `us-east-1`); cross-region adds latency and complexity without benefit for a single-region app.
| Service quotas | Default Fargate vCPU/memory quotas are sufficient for a small deployment; request an increase in the Service Quotas console before onboarding many customers/environments.
|===

=== Local / CI tooling

[cols="1,3", options="header"]
|===
| Tool | Why
| `aws` CLI v2 | Drives every command in this guide (`aws ecs`, `aws ecr`, `aws rds`, …). Configure credentials with `aws configure` or an SSO profile.
| Docker (with Buildx) | Builds the same image used by the Kamal guide — `docker build -t capri-dam .` from the repository root.
| `jq` (optional) | Used in a few playbook commands to extract values from `aws` CLI JSON output.
|===

Verify you're authenticated to the right account before continuing:

[source,bash]
----
aws sts get-caller-identity
----

=== DNS

You need a domain (or subdomain, e.g. `dam.example.com`) whose zone you can edit — either delegate it
to **Route 53** (recommended, simplifies ACM DNS validation) or keep it at your existing registrar and
add a CNAME later pointing at the ALB's DNS name. See
<<section-aws-ssl-domains-and-routing,SSL, Domains & ALB Routing>> for the exact records.

=== Networking decision: use an existing VPC or create a new one

This guide creates a new, dedicated VPC (two public + two private subnets across two Availability
Zones) so the playbook is self-contained and doesn't risk touching existing infrastructure. If you
already have a VPC with private subnets and a NAT gateway, skip straight to
<<section-aws-infrastructure-reference,Infrastructure Reference>> and substitute your existing subnet
IDs/security groups throughout.

[cols="1,3", options="header"]
|===
| Subnet tier | Purpose
| Public (2× AZ) | ALB only. Has a route to an Internet Gateway.
| Private (2× AZ) | ECS tasks, Aurora, ElastiCache. Route to a NAT Gateway for outbound (image pulls from ECR, gem/npm-style calls if any, AWS API calls) — no direct inbound internet access.
|===

=== Cost note

A NAT Gateway (~$0.045/hr + data processing) and Aurora/ElastiCache are the main fixed costs beyond
Fargate vCPU/memory-seconds. For a low-traffic staging environment, a single NAT Gateway (not one per
AZ) and Aurora Serverless v2 (scales to near-zero) meaningfully reduce idle cost — both are called out
inline in <<section-aws-infrastructure-reference>> and <<section-aws-managed-data-services>>.
