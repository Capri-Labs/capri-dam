== Secrets & Credentials

=== How `.kamal/secrets` works

`.kamal/secrets` (committed to this repo — see the file itself for why that's safe) never
contains a real secret value. Each line only references an environment variable that must already
be set **on the machine invoking `kamal deploy`** (your laptop, or a CI runner):

[source,bash]
----
KAMAL_REGISTRY_USERNAME=$KAMAL_REGISTRY_USERNAME
KAMAL_REGISTRY_PASSWORD=$KAMAL_REGISTRY_PASSWORD
RAILS_MASTER_KEY=$RAILS_MASTER_KEY
HEADLESS_DAM_DATABASE_PASSWORD=$HEADLESS_DAM_DATABASE_PASSWORD
POSTGRES_PASSWORD=$HEADLESS_DAM_DATABASE_PASSWORD
----

At deploy time, Kamal reads this file, resolves each `$VARIABLE` against your **local** shell
environment, and ships the resolved values to the remote server(s) over SSH — encrypted in
transit, injected directly into the container's environment, never written to disk on the server
in plaintext, and never logged.

=== Populating the local/CI environment

Two supported approaches:

==== Option A — a git-ignored `.env` file (simplest)

This repo already uses `dotenv-rails` for development (`gem "dotenv-rails", groups: [:development,
:test]`) and already git-ignores `.env*` (see `.gitignore`, with `.env.example` explicitly
allow-listed as the checked-in template). Create a **local-only** `.env` (or `.env.deploy`, loaded
manually) with:

[source,bash]
----
KAMAL_REGISTRY_USERNAME=your-github-username
KAMAL_REGISTRY_PASSWORD=ghp_yourPersonalAccessTokenWithPackagesScope
RAILS_MASTER_KEY=$(cat config/master.key)
HEADLESS_DAM_DATABASE_PASSWORD=a-long-random-password
KAMAL_WEB_HOST=203.0.113.10
KAMAL_APP_HOST=dam.example.com
----

Then export it before running any `kamal` command:

[source,bash]
----
set -a; source .env; set +a
bundle exec kamal deploy
----

==== Option B — CI secrets (recommended for team use)

Store the same variables as encrypted secrets in your CI provider (e.g. GitHub Actions
repository/environment secrets) and export them as environment variables in a deploy job step
before invoking `bundle exec kamal deploy`. This repo's `.github/workflows/docker.yml` already
demonstrates the pattern for `RAILS_MASTER_KEY`; a deploy workflow would add the additional
`KAMAL_*` / `HEADLESS_DAM_DATABASE_PASSWORD` secrets the same way.

==== Option C — a secrets manager (1Password, etc.)

Kamal has first-class support for resolving secrets from 1Password via `kamal secrets extract`
helpers instead of plain environment variables. Not configured in this starter setup (to keep the
barrier to a first deploy low), but `.kamal/secrets` can be swapped for 1Password-backed lookups
later without touching `config/deploy.yml` — see the upstream docs:
https://kamal-deploy.org/docs/configuration/secrets/.

=== What each secret is for

[cols="1,3", options="header"]
|===
| Variable | Used for
| `KAMAL_REGISTRY_USERNAME` / `KAMAL_REGISTRY_PASSWORD` | Authenticate to `ghcr.io` to pull (and optionally push) the app image. A GitHub Personal Access Token with `read:packages` (and `write:packages` if building/pushing locally) works.
| `RAILS_MASTER_KEY` | Decrypts `config/credentials.yml.enc` at boot — identical to `config/master.key`'s content. **Never** commit `config/master.key` itself (already git-ignored).
| `HEADLESS_DAM_DATABASE_PASSWORD` | The app's Postgres connection password (`config/database.yml` → `production.primary.password`).
| `POSTGRES_PASSWORD` | The same value, passed to the `db` accessory container so the accessory's own password matches what the app connects with (see <<section-accessories,Accessories>>).
|===

=== Rotating a secret

1. Generate a new value (e.g. a new database password).
2. Update it wherever it's sourced from (`.env`, CI secret store, or 1Password item).
3. Re-run `kamal deploy` (app containers) — new `env.secret` values are picked up automatically.
4. For `POSTGRES_PASSWORD`, you must also update the password *inside* Postgres itself
   (`ALTER USER headless_dam WITH PASSWORD '...'` via `kamal accessory exec db -- psql ...`)
   before redeploying the app, or the app will fail to authenticate — see
   <<section-accessories,Accessories>> for the exact command.
