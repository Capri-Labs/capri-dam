= Duplicate Detection

Duplicate detection uses two independent, complementary strategies, both
implemented in `DuplicateDetectionService`:

[cols="1,3", options="header"]
|===
| Strategy | Notes
| Exact match | Groups assets by SHA-256 checksum (`AssetVersion.properties["checksum_sha256"]`). Catches byte-identical files (same format, same bytes).
| Perceptual match | Groups assets by a 64-bit perceptual hash (dHash, `AssetVersion.properties["perceptual_hash"]`), computed by `AssetProcessorWorker#extract_perceptual_hash` via ImageMagick during ingestion. Catches the *same photo saved in different formats* (e.g. `.png` vs `.jpg`), which have different bytes/checksums but are visually identical. Two hashes are considered a match when their Hamming distance is `<= DuplicateDetectionService::PERCEPTUAL_HAMMING_THRESHOLD` (10 out of 64 bits).
|===

Perceptual-match groups are stored in the same `duplicate_groups.checksum`
column as exact matches, but prefixed `"phash:<hash>"` to avoid colliding with
real SHA-256 values (no schema migration required). Exact matches always take
priority — an asset already grouped by checksum is not also re-grouped
perceptually.

== Detection flow

[cols="1,3", options="header"]
|===
| Stage | Notes
| Repository scan | `DuplicateRepositoryScanWorker` walks the library and tracks progress in `Setting`.
| Grouping | `DuplicateDetectionService` groups matching checksums (or near-matching perceptual hashes) into `DuplicateGroup` rows.
| Resolution | Users keep all members, delete duplicates, or dismiss a false positive.
|===

== Group lifecycle

[cols="1,3", options="header"]
|===
| Status | Meaning
| `pending` | Needs review.
| `resolved` | Canonical result recorded.
| `dismissed` | False positive or intentionally ignored.
|===

A group must have **2+ active (non-trashed) members** to remain meaningfully
"pending" — falling below that threshold auto-resolves it so the Duplicate
Manager never shows a stale "1 asset found" group. This is enforced
automatically from two independent code paths so no caller has to remember
to do it manually:

* **Soft-delete / restore** (moved to or recovered from the Trash Bin) —
  `Asset` has an `after_commit :sync_duplicate_groups_membership, if: :saved_change_to_deleted_at?`
  hook that calls `DuplicateGroup#recalculate_active_membership!` for every
  group the asset belongs to. This recomputes `total_count` from
  `Asset.active` members and:
** auto-resolves the group (`status: "resolved"`, `resolution_action: nil` —
   distinguishing it from a user-driven resolution) once fewer than 2 active
   members remain;
** reopens a previously *auto*-resolved group back to `pending` if a restore
   brings the active member count back to 2+ (a group resolved by explicit
   user action, i.e. `resolution_action` present, is never silently reopened).
* **Hard/permanent deletion** — `DuplicateGroupAsset` has an
  `after_destroy :auto_resolve_group_if_depleted` callback that fires
  whenever a membership row is destroyed by *any* means: the explicit
  `DuplicateGroupAsset.cleanup_for_asset!` helper (called from
  `Api::V1::AssetsController#permanent_delete`, `BinPurgeService`, and
  `EmptyBin`), or an incidental `dependent: :destroy` cascade from
  `Asset#destroy!` (e.g. via `Folder#destroy`). This is a safety net that
  guarantees correctness even if a future code path hard-deletes an asset
  without remembering to call `cleanup_for_asset!` first.

== APIs

[cols="1,2,3", options="header"]
|===
| Method | Path | Description
| `GET` | `/api/v1/duplicate_groups` | List groups by status.
| `GET` | `/api/v1/duplicate_groups/stats` | Count summary.
| `GET` | `/api/v1/duplicate_groups/:id` | Group detail with member assets.
| `PATCH` | `/api/v1/duplicate_groups/:id/resolve` | Resolve a group.
| `PATCH` | `/api/v1/duplicate_groups/:id/dismiss` | Dismiss a group.
|===

GraphQL also exposes `duplicateGroups`, `duplicateManagerStats`,
`duplicateManagerScanStatus`, `resolveDuplicateGroup`, and
`triggerDuplicateScan`.

`POST /api/v1/assets/check_hashes` is a separate, upload-time check (distinct
from the `DuplicateGroup` repository scan above): it takes a batch of
client-computed file hashes and returns any matching existing `AssetVersion`
records per hash, keyed for `Folders/UploadWorkspace.jsx`'s duplicate-overlay
flow. Each matched entry includes
`in_bin: asset.deleted_at.present? || asset.folder&.deleted_at.present?` —
true when either the asset itself was soft-deleted *or* its containing folder
was (an asset whose folder is trashed is only reachable via the Bin, so it
must be flagged the same way) — so the upload-time comparison UI
(`DuplicateResolverDialog.jsx`) can flag a match that is currently in the
Recycle Bin with the same `BinChip` badge Search results use — see
<<section-ui-components>>.

== UI

`DuplicateManager.jsx` provides filter chips, stats cards, detail fetching, and
launches `DuplicateResolutionModal` for destructive review decisions.
`DuplicateResolutionModal`'s "Go to Folder" / "Go to asset" actions navigate to
`/folders?folder=<id>` and `/assets?id=<id>` respectively — matching the query
params `AssetExplorer.jsx`'s `readUrlFilters()` expects (folder navigation
uses `?folder=`, asset deep-linking uses `?id=`; they are intentionally
different params).

The resolution modal's "Dismiss" icon button carries an explicit
`aria-label` (in addition to its `Tooltip` title) so it has an accessible
name for screen readers and can be targeted directly in tests, matching the
pattern already used by the "Go to asset"/"Go to Folder" icon buttons in the
same file.

  `spec/e2e/search_and_duplicates_fixes.e2e.spec.js`'s two navigation
  regressions):
** Duplicate Manager page loads and lists existing duplicate groups
   (including the empty-state message when there are none).
** The side-by-side resolution modal opens and shows both duplicate members.
** Resolving a group — either "Keep All" or deleting a selected copy — closes
   the modal, removes the group from the pending queue, and moves it to the
   "Resolved" filter.
** Dismissing a group excludes it from future pending prompts.
