[[section-models]]
= Domain Models

Capri DAM uses conventional Rails models under `app/models/`, but several
subsystems add domain-specific state machines, JSONB-backed configuration, and
join models that are important for extension work.

== Core asset and library models

=== Asset-adjacent records

[cols="1,1,3", options="header"]
|===
| Model | Key fields | Notes

| `AssetVersion`
| `version_number`, `file_size`, `properties`
| Immutable snapshot of a binary revision. The active asset keeps the current
master; historical revisions remain addressable for restore, checksum analysis,
and duplicate detection.

| `Rendition`
| `kind`, `file`, `metadata`
| Cached derivative output such as thumbnails or transformed delivery assets.
The platform still prefers dynamic CDN/image-IO delivery where possible.

| `AssetEmbedding`
| `embedding`, `model_name`, `status`
| Stores the pgvector payload used by semantic search, smart collections,
cluster-map visualisation, and Copilot-style discovery.

| `AssetProvenanceRecord`
| `manifest_status`, `is_ai_generated`, `is_ai_modified`, `verification_details`
| Normalised C2PA/provenance result for an asset. Populated from gateway
callbacks and queried by provenance dashboards.

| `AssetWorkflowTrigger`
| runtime-only association via `workflow_instances`
| Assets do not store workflow logic directly; matching is delegated to
`AssetWorkflowTriggerWorker` and persisted as `WorkflowInstance` rows.
|===

=== Folder-browsing performance (large folders)

Opening a folder with 1,000-3,000+ assets is the most expensive read path in
the app: `FoldersController#show` builds a fully-formatted listing (URLs,
thumbnails, sort) for every active asset in the folder on every request.

* **`FolderContentsCache`** (`app/services/folder_contents_cache.rb`) — a
  Redis-backed cache mirroring `SearchCache` (see
  <<section-search,`09_search.adoc`>>): namespaced keys
  (`dam:folder_contents:<folder_id>:<params-hash>`), 30s TTL, defensive
  `Redis::BaseError`/`SystemCallError` fallback to uncached execution, and
  skipped entirely in the test environment. Permission checks in
  `FoldersController#show` always run **uncached**, before the cache is
  consulted, so a cache hit never bypasses authorization.
* **Active busting**, not just TTL expiry: `FolderContentsCache.bust` is
  called on every mutation that can change a folder's listing — folder
  create/update/destroy/restore (busts the affected `parent_id`), asset
  create/update/destroy/restore/permanent_delete/update_metadata (busts the
  old and new `folder_id` on a move), and `ApplySchemaToFolderJob` (busts
  each folder level it cascades through). Bulk operations in
  `BinController` (`bulk_restore`, `bulk_destroy`, `empty`) call
  `FolderContentsCache.flush_all` instead of tracking individual folder ids,
  since a bulk op can reparent/touch arbitrarily many folders at once.
* Renaming/moving a folder busts its **parent's** cache (the parent's
  listing shows this folder's name/position) — not the folder's own cache,
  since a folder's own children listing is unaffected by its own name
  change.
* **Root-level `nil` `folder_id` bug (fixed)**: `FolderContentsCache.bust`
  used to do `Array(folder_id).uniq.each { ... }`. Ruby's `Kernel#Array`
  collapses a bare `nil` to `[]` (not `[nil]`), so `bust(nil)` — the call
  every root-level folder/asset create, rename, delete, and restore makes —
  silently iterated zero times and never invalidated the root listing's
  cache entries. The practical symptom: creating/renaming/deleting anything
  at the top level could leave the `/folders` root view stale for up to the
  30s TTL. Fixed by wrapping explicitly
  (`folder_id.is_a?(Array) ? folder_id : [folder_id]`) so a bare `nil`
  correctly busts the `root` namespace. Regression-covered in
  `spec/services/folder_contents_cache_spec.rb` (asserts `redis.scan`/`del`
  are actually invoked for the `nil` case, not just that no error is raised).
* **N+1 fix**: `AssetUrlHelper#asset_url_for` used to call
  `StorageBackend.find_by(active: true)` once per asset formatted — for a
  3,000-asset folder that's 3,000+ redundant identical queries per request.
  It is now memoized per-request via a private `active_storage_backend`
  helper (safe because concerns are mixed into a fresh controller instance
  per request).
* **Indexes**: `db/migrate/20260709163140_add_performance_indexes_for_large_folders.rb`
  adds composite indexes on `assets(folder_id, deleted_at)` and
  `folders(parent_id, deleted_at)` (the two columns every folder listing
  filters/sorts by), plus expression indexes on the JSONB `properties` keys
  with fixed, hardcoded call sites (`content_type`, `applied_schema_id`,
  `file_size`). The existing whole-column GIN index on `properties` only
  accelerates containment operators (`@>`/`?`/`?&`/`?|`); it does not speed
  up `properties->>'key'` extraction, which is what these hot filters use.
  With 100+ possible schema-defined metadata keys queried dynamically via
  `apply_dynamic_filters`, it is not feasible to statically index every
  key — only fixed call sites got dedicated indexes; add a matching
  expression index for a specific custom field later if it becomes a hot
  filter path. The `file_size` index is a **partial** index
  (`WHERE (properties->>'file_size') ~ '^[0-9]+$'`) because Postgres
  evaluates expression indexes on every write, and an unconditional
  `::bigint` cast raises `PG::InvalidTextRepresentation` on legacy
  non-numeric values.

=== Access, identity, and auditing

[cols="1,1,3", options="header"]
|===
| Model | Key fields | Notes

| `AuditLog`
| `auditable_type`, `auditable_id`, `action`, `changeset`, `actor_id`
| Append-oriented change ledger used for operational traceability, admin
history, and compliance review.

| `Notification`
| `title`, `message`, `read_at`, `action_url`
| Lightweight in-app notification shown by `NotificationBell.jsx`. Created by
workflow/task flows and cleared through `/api/v1/notifications`.

| `PersonalAccessToken`
| `name`, `token_digest`, `last_used_at`, `expires_at`, `revoked_at`
| Token-based auth for automation. Tokens are created once, stored hashed, and
can be revoked without touching the owning `User`.

| `Setting`
| `key`, `value`
| Global key/value store used for operational toggles such as duplicate scan
status, bin retention, and health-check markers.

| `SystemConfiguration`
| `logging_level`, `retention_days`, feature flags in JSONB
| Runtime operational configuration edited in the admin module.

| `SystemConnector`
| `provider_type`, `endpoint`, `status`, `tdm_sanitation`, `analysis_report`
| Source-system connection used by migrations and ingestion. Tracks health,
asset counts, and webhook/migration metadata.
|===

== Collections, duplicates, and governance

=== Collections

`Collection` is richer than a simple tag set. The model combines curated and
rule-driven groupings, temporal visibility, and attribute-based access controls.

[cols="1,1,3", options="header"]
|===
| Model | Key fields | Notes

| `Collection`
| `collection_type`, `slug`, `expires_at`, `properties`
| Supports manual and smart workspaces. `properties` stores governance fields
such as `allowed_groups`, `denied_groups`, and other presentation settings.
Expired rows are filtered from the `active` scope.

| `CollectionRule`
| `semantic_prompt`, `similarity_threshold`, `rule_type`
| Smart-collection routing rule. The current implementation is semantic and
prompt-driven rather than classic `field/op/value` filtering.

| `CollectionAsset`
| `collection_id`, `asset_id`, `position`, `pinned`
| Join row for manual membership plus pinning/order information used by the UI.

| `DuplicateGroup`
| `checksum`, `status`, `resolution_metadata`
| SHA-256 based duplicate bucket. Lifecycle is `pending`, `resolved`, or
`dismissed`; resolution metadata records which asset remained canonical.
|===

=== Governance and delivery configuration

[cols="1,1,3", options="header"]
|===
| Model | Key fields | Notes

| `C2paConfiguration`
| `gateway_c2pa_enabled`, `auto_verify_on_ingest`, `verification_strictness`
| Organisation-wide provenance policy consumed by `AssetProvenanceWorker` and
admin provenance tooling.

| `CdnConfiguration`
| provider credentials, signing settings, purge defaults
| Canonical CDN/edge configuration backing `CdnManager` and the edge-operation
API.

| `StorageBackend`
| `service_name`, `provider`, `bucket`, `active`
| Admin-managed storage target. Used when switching local/S3/GCS/Azure-style
backends without changing code.

| `QuarantinedAsset`
| review metadata, source asset linkage
| Records binaries isolated from normal delivery because of validation,
security, or ingestion-policy issues.

| `FolderPolicy`
| `read_access`, `modify_access`, `create_access`, `delete_access`, `replicate_access`, `manage_access`, `explicit_deny`
| The current ACL matrix used by admin and folder APIs. This is broader than the
older `can_*` naming still visible in stale docs.
|===

== Ingestion and metadata operations

=== Ingestion pipeline records

[cols="1,1,3", options="header"]
|===
| Model | Key fields | Notes

| `IngestionBatch`
| `source_type`, `status`, `processed_count`, `duplicate_count`, `error_count`
| Migration/import batch state machine. Observed states are `initializing`,
`extracting`, `transforming`, `review_needed`, `committed`, and `failed`.

`rejected`.
