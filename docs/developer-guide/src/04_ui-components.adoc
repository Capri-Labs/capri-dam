`app/javascript/components/`. Complex modules are mounted as islands inside the
Rails shell and typically fetch JSON from `/api/v1/...`.

== Version history comparison

`Folders/AssetVersionsTab.jsx` now supports a two-version compare mode for asset
history. The default timeline remains unchanged, but users can select any two
versions and open a comparison workspace that renders:

* a blended before/after stack for quick inspection,
* a Canvas-computed pixel-diff overlay highlighted in magenta, and
* separate before/after previews for side-by-side review.

The diff implementation is intentionally dependency-free: both preview URLs are
loaded into padded canvases sized to the larger source image, then the component
computes a simple per-pixel absolute RGBA delta to build the overlay canvas.
When browser security prevents `getImageData()` access for cross-origin/CDN
images, the component catches the `SecurityError` and falls back to plain
side-by-side previews instead of breaking the Versions tab.

[[section-rename-overlay]]
== Rename overlay (folders & assets)

Single-item rename is offered from the Explorer's *Tools* menu
(`ExplorerTopBar.jsx`) whenever the selection is exactly **one** folder XOR
**one** asset (never both, never multiple — the menu item is entirely
absent, not just disabled, for any other selection shape).

[cols="1,3", options="header"]
|===
| Piece | Notes
| `Folders/RenameDialog.jsx` | Single-target dialog (folder or asset), modeled on `ApplySchemaDialog.jsx`'s conventions. `PATCH /api/v1/folders/:id` with `{ folder: { name } }`, or `PATCH /api/v1/assets/:id` with `{ asset: { title } }`. Validates the new name is non-empty and actually changed (Save stays disabled otherwise), surfaces server-side validation/permission errors inline without closing, and notifies via `useNotify` on success.
| `ExplorerTopBar.jsx` — `renameTarget` resolution | Computed from `selectedItems`/`viewData`: a folder is only offered when `selectedItems.folders.length === 1 && selectedItems.assets.length === 0` (and symmetrically for a single asset). `canModify` reads the `can_modify` flag returned by the folder-contents payload, treated as **permissive by default** (`item.can_modify !== false`) so older/mocked payloads without the field still show the option — consistent with the existing `editable ?? isNativelyRenderableImage` pattern used elsewhere.
| Refresh wiring | Reuses the existing `onSchemaApplied` callback (already wired to `loadContent()` in `AssetExplorer.jsx`) for `RenameDialog`'s `onClose(needsRefresh)` — no new prop plumbing needed.
|===

=== Permission model — `can_modify`

`Api::V1::FoldersController#format_folder_payload` /
`#format_asset_payload` both include a `can_modify` boolean, backed by a new
private `can_modify_folder?(folder)` helper that mirrors
`Authorization#check_asset_modify!`'s semantics:

* Admins: always `true`.
* `folder.nil?` (root-level items): always `true` — no `FolderPolicy` row can
  apply at the root, so there's nothing to deny.
* Otherwise: `!!current_user&.permissions_for(folder)&.dig(:modify)`.

`FoldersController#update` (the rename endpoint) previously had **no
permission check at all** — any authenticated user could rename any folder.
It now calls `check_folder_permission!(@folder, :modify)` before applying
changes, matching `AssetsController#update`'s existing `check_asset_modify!`
guard.

CAUTION: `User#permissions_for(folder)` checks only the **exact** folder's
`FolderPolicy` rows — it does not walk the ancestor chain. Granting
`modify_access` on a parent folder does **not** automatically grant
`can_modify: true` on its subfolders; each folder's flag is evaluated
against its own `folder_id`. This is pre-existing, unchanged behaviour, but
worth knowing when writing specs/fixtures that expect inherited access.

[[section-move-overlay]]
== Move overlay (folders & assets)

Bulk move is offered from the Explorer's *Tools* menu (`ExplorerTopBar.jsx`)
whenever the selection contains **one or more** folders and/or assets (mixed
selections are allowed, unlike Rename).

[cols="1,3", options="header"]
|===
| Piece | Notes
| `Folders/MoveDialog.jsx` | Multi-target dialog. Fetches `GET /api/v1/folders` to populate a searchable MUI `Autocomplete` of every active folder (already returns full breadcrumb `name` paths), plus a synthetic root (`{ id: 'root', name: '/', isRoot: true }`) option, excluding the folder(s) being moved themselves from the candidate list client-side (the server still rejects self/descendant moves — see cycle-prevention below — for the cases the client-side filter can't catch, e.g. moving into a nested descendant). Submits the whole batch in one request: `POST /api/v1/move_operations` with `{ folder_ids, asset_ids, destination_folder_id }`. Renders a per-item error list (`itemErrors`) for partial failures without discarding the whole dialog.
| `ExplorerTopBar.jsx` — `moveTarget` resolution | Computed whenever `selectedItems.folders.length + selectedItems.assets.length >= 1`. `canDelete` is `true` only when *every* selected item's `can_delete` flag is not explicitly `false` (`item.can_delete !== false`, permissive-by-default like `canModify`). The menu item itself is always shown for any non-empty selection but rendered `disabled` when `canDelete` is `false`, with a `moveNoPermission` helper string explaining why.
| Refresh wiring | `MoveDialog`'s `onClose(needsRefresh)` triggers the same `loadContent()` refresh callback used by Rename/Upload.
|===

=== Permission model — Move = delete-on-source + create-on-destination

Moving is modeled as *removing* an item from its current folder and
*adding* it to another, so it is gated by **both** halves of that operation
rather than by a single `modify`/`move` permission bit:

* `folder_permission?(source_folder, :delete)` — the user must be able to
  delete from wherever the item currently lives.
* `folder_permission?(destination_folder, :create)` — the user must be able
  to create/add items in the destination.
* `folder.nil?` (root) always passes both checks — no `FolderPolicy` row can
  apply at the root level.
* Admins bypass both checks.

This is enforced identically across three surfaces so there is exactly one
way to move something without the right permissions being rejected:

* `FoldersController#update` — when `parent_id` is present in the payload
  *and differs* from the folder's current `parent_id` (a plain rename with an
  unchanged `parent_id` is unaffected and still only requires `modify`).
* `AssetsController#update` — same rule, keyed on `folder_id`.
* `Api::V1::MoveOperationsController#create` — the new bulk endpoint; each
  folder/asset in the batch is checked independently, so one item failing
  permission/cycle validation doesn't abort the rest (partial success, with
  per-item errors returned in the `errors` array).

`Authorization#folder_permission?(folder, permission)` is a new non-halting
predicate extracted from the existing `check_folder_permission!` (which now
delegates to it and additionally renders `403` on failure) — needed because
bulk Move must evaluate permission per-item without aborting the whole batch
on the first denial.

=== Cycle prevention

`Folder#self_or_ancestor_match?(candidate_id)` walks up the ancestor chain
starting at `candidate_id` looking for `self`, returning `false` immediately
for a `nil` candidate (moving to root is never a cycle). Both
`FoldersController#update` and `MoveOperationsController#create` call this
before applying a folder move and reject with `422` when a folder would be
moved into itself or one of its own descendants.

[[section-copy-overlay]]
== Copy overlay (folders & assets)

Bulk copy sits directly below Move in the same *Tools* menu
(`ExplorerTopBar.jsx`), offered under the identical selection rule — **one or
more** folders and/or assets, mixed selections allowed.

[cols="1,3", options="header"]
|===
| Piece | Notes
| `Folders/CopyDialog.jsx` | Near-identical sibling of `MoveDialog.jsx` — same folder-picker `Autocomplete` (fed by `GET /api/v1/folders`), same synthetic root option, same per-item `itemErrors` rendering for partial failures. Submits `POST /api/v1/copy_operations` with `{ folder_ids, asset_ids, destination_folder_id }` in one request.
| `ExplorerTopBar.jsx` — `copyTarget` resolution | Computed under the same `moveItemCount > 0` condition as `moveTarget`, but with **no client-side permission gate** — copying never removes anything from the source, so unlike Move there is no `can_delete`-equivalent flag to check before opening the dialog; the only real gate (destination `:create`) can't be evaluated until a destination folder is picked, so it is left entirely to the server response, exactly like Move's own destination check already is.
| Refresh wiring | `CopyDialog`'s `onClose(needsRefresh)` triggers the same `loadContent()` refresh callback used by Move/Rename/Upload.
|===

=== Permission model — Copy = read-on-source + create-on-destination

Copying duplicates an item rather than relocating it, so it needs one fewer
check than Move — nothing is ever removed from the source:

* `folder_permission?(source_folder, :read)` — the user must be able to view
  wherever the item currently lives (in practice this is already implied by
  the item being visible in the Explorer listing the user is copying from).
* `folder_permission?(destination_folder, :create)` — the user must be able
  to create/add items in the destination.
* `folder.nil?` (root) always passes both checks.
* Admins bypass both checks.

`Api::V1::CopyOperationsController#create` is the sole surface for bulk copy
(there is no "implicit copy" analogous to Move's `parent_id`/`folder_id`
change on `FoldersController#update`/`AssetsController#update` — copying is
only ever an explicit, additive operation). Each folder/asset in the batch is
evaluated independently, exactly like Move, so partial success is possible
with per-item errors reported in the `errors` array.

=== Recursive duplication & name-collision handling

* Copying a **folder** recursively duplicates its entire active subtree —
  every descendant subfolder and asset is duplicated alongside it — rooted
  under a single new top-level folder in the destination.
* Copying an **asset** duplicates its metadata (`properties`, `status`) plus
  its active version's stored file. The new `AssetVersion`
  (`action_type: "copy"`) gets its own physical file, copied at the
  `storage_path` layer via `StorageManager.active_adapter.copy` — the same
  provider-agnostic mechanism (local disk, S3, GCS, ...) the upload/processing
  pipeline uses. `storage_path` is deliberately preferred over the legacy
  ActiveStorage attachment: `active_storage_attachments.record_id` is a
  `bigint` column but `Asset`/`AssetVersion` use UUID primary keys, so every
  attachment's `record_id` is coerced to `0` and `file.attached?` can
  spuriously match a *different* record's blob. Falls back to sharing the
  ActiveStorage blob only when a version has no `storage_path` at all (e.g.
  legacy/seed data).
* All copies are owned by the requesting user (`current_user`), matching the
  fact that a copy is genuinely new content gated by the destination's
  `:create` check, not a transfer of the original's ownership.
* Name collisions in the destination are resolved automatically —
  `" (Copy)"`, `" (Copy 2)"`, … — rather than failing the request. The
  original name is preserved untouched when there's no collision (copying to
  a different folder), matching familiar OS file-manager behaviour.
* `Folder#self_or_ancestor_match?` is reused unchanged to reject copying a
  folder into itself or one of its own descendants (`422`), identically to
  Move's cycle guard.

== Upload & Enrich overlay

`Folders/UploadWorkspace.jsx` is the full-screen upload dialog launched from the
"Upload Asset" button in `ExplorerTopBar.jsx`. It composes `UploadSidebar.jsx`
(global assignment/tags/AI actions) and `UploadGrid.jsx` (the per-file staging
grid), and drives duplicate detection via `DuplicateResolverDialog.jsx`.

[cols="1,3", options="header"]
|===
| Behaviour | Notes
| Tag merging on upload | `handleUploadAll()` sends a single de-duplicated `tags` metadata array that merges the sidebar's global `manualTags` (typed into the Tags `Autocomplete`, `data-testid="upload-manual-tags"`) with each file's own `meta.aiTags` (set per-file by the "Run AI enhance on this file" button in `UploadGrid.jsx`). Both sources previously landed in different, disconnected pieces of state — only the sidebar's manual tags were actually sent to `POST /api/v1/assets`, so AI tags shown in the staging grid never made it into `Asset#properties.tags` and never appeared in `AssetTagsEditor.jsx`/`AssetViewer.jsx` afterward. `data-testid="upload-global-asset-type"` marks the "Select Global asset type" `Select` for e2e targeting.
| Duplicate detection at upload time | Dropped files are hashed client-side (`calculateFileHash`) and checked in a single batch via `POST /api/v1/assets/check_hashes`. A match sets `isDuplicate: true` on that file and shows an amber "Duplicate Found" chip in `UploadGrid.jsx`, opening `DuplicateResolverDialog.jsx`'s `LegacyDuplicateResolverDialog` for side-by-side comparison and resolution (Skip / Upload anyway / AI merge for identical bytes).
| Bin-aware duplicate matches | `check_hashes` returns `in_bin: asset.deleted_at.present? \|\| asset.folder&.deleted_at.present?` per matched duplicate (see <<section-duplicate-detection>>) — true when the asset itself is soft-deleted *or* its containing folder is (an asset inside a trashed folder is only reachable via the Bin). `LegacyDuplicateResolverDialog` renders the same `BinChip` pattern used by Search results (`SearchResultCard.jsx`) as an overlay badge on the primary comparison image, and next to any per-location chip, whenever the matched existing asset currently lives in the Recycle Bin — so a user can't be confused into thinking a soft-deleted asset (or one whose folder was binned) is still active in the library.
|===

== Asset Viewer — Info tab

`Folders/AssetViewer.jsx`'s Info tab (`TabPanel` index 0) renders the curated
general-metadata list plus two independently-collapsible raw-JSON debug
panels, both collapsed by default so the panel doesn't dump a large blob into
view on open:

[cols="1,3", options="header"]
|===
| Panel | Notes
| EXIF / IPTC / XMP Data | Controlled by the `exifExpanded` state. Renders `asset.properties.embedded_metadata` (falling back to the legacy `asset.properties.exif_data`) as pretty-printed JSON inside a scrollable `Paper`. Toggle header/icon `aria-label` flips between `assetViewer.info.expandExifData` / `assetViewer.info.collapseExifData`.
| Raw Metadata | Controlled by the separate `rawMetadataExpanded` state, rendered directly below the EXIF panel (behind its own `Divider`). Renders the asset's *entire* raw `properties` JSONB blob (not just the embedded-metadata subset) as pretty-printed JSON, so any field stored on the asset — including ones with no dedicated UI row — can be inspected. Falls back to `assetViewer.info.noRawMetadataAvailable` when `properties` is empty. Toggle header/icon `aria-label` flips between `assetViewer.info.expandRawMetadata` / `assetViewer.info.collapseRawMetadata`. Expanding/collapsing this panel is independent of the EXIF panel's state.
|===
