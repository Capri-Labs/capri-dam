= Admin Module

The Admin module is a set of Rails-rendered shells backed by React islands under
`app/javascript/components/Admin/`. It centralises identity, access control,
infrastructure, messaging, migration operations, and system observability.

== User and group administration

`UsersManager.jsx`, `UserDrawer.jsx`, and `UserGroupsManager.jsx` are the main
screens for account lifecycle work.

[cols="1,3", options="header"]
|===
| Capability | Implementation notes
| User management | Create, edit, disable, and inspect users; assign roles and group memberships.
| Group hierarchy | Nested `UserGroup` trees support inherited membership and permission roll-up.
| Drawer detail | `UserDrawer.jsx` surfaces per-user metadata, memberships, and admin actions without leaving the list.
| Group pagination | `UserGroupsManager.jsx` paginates root-level custom groups 10/page (`ROOT_GROUPS_PER_PAGE`); a group's descendants always render alongside it regardless of page, preserving tree integrity.
| Group bulk delete | Root-level custom groups on the current page can be multi-selected and deleted via `DELETE /admin/user_groups/bulk_delete` (`ids: []`); system groups are never selectable and are skipped/reported (`skipped_system_ids`) if included.
|===

== ACL matrix and folder policy administration

`AclMatrix.jsx` edits the current folder-policy columns directly.

[cols="1,1,3", options="header"]
|===
| Field | Type | Description
| `read_access` | boolean | Browse folder and asset content.
| `modify_access` | boolean | Edit metadata/content.
| `create_access` | boolean | Upload or create children.
| `delete_access` | boolean | Trash items.
| `replicate_access` | boolean | Copy/share/export operations.
| `manage_access` | boolean | Folder administration.
| `explicit_deny` | boolean | Hard deny that overrides permissive inheritance.
|===

=== Collapsible folder tree, cascade edit, and cascade save

`AclMatrix.jsx` builds an in-memory folder tree from the `parent_id` field
returned by `GET /api/v1/folders.json` (see
<<section-api-reference,API reference>>). Only root folders (`parent_id ==
null`) render by default; a folder with children shows a chevron toggle that
reveals its immediate children in place, and "Expand all" / "Collapse all"
icon buttons are available in the toolbar. The table body scrolls
(`maxHeight: 520px`) so large folder trees stay compact. Searching the
"Filter folders…" box switches to a flat, fully-matched list regardless of
tree position.

Two independent cascade behaviours exist and can be combined:

* *Client-side toggle cascade* — toggling any permission (or Deny All) on a
  folder immediately copies the new matrix onto every currently-loaded
  descendant row, so bulk edits across a subtree are fast. Each descendant
  remains individually editable/overridable — unchecking one afterwards
  simply reverts that folder's local state.
* *Server-side save cascade* — a per-folder "Also save to subfolders" switch
  (visible only on folders with children) sends `cascade: true` in the
  `POST /admin/folders/:folder_id/folder_policies` body. When set,
  `Admin::FolderPoliciesController#create` enqueues
  `PropagateAccessPolicyJob`, which recursively upserts the same permission
  matrix onto every descendant `Folder` in the database (mirroring the
  cascade already used by `Api::V1::FoldersController#upsert_folder_policy`
  for the per-folder Access tab).

== Operational configuration


`SystemConfigurationsController` and the System Status UI provide runtime
controls instead of compile-time environment switching.

[cols="1,3", options="header"]
|===
| Area | Notes
| System configuration | Log level, TTL-style retention knobs, and feature toggles.
| Storage backend admin | Manage `StorageBackend` definitions and active targets.
| Email engine | `EmailEngineManager.jsx` manages templates, deliveries, and SMTP-related workflows.
| Report snapshots | Historical `ReportSnapshot` files are available from the reports area.
|===

== System status

`Admin/SystemStatus/index.jsx` exposes operational tabs for:

* AI gateway connectivity and feature availability
* SMTP configuration and test-send
* observability/runtime diagnostics
* Redis/Sidekiq and background queue health
* storage operations and service checks
* operational logging / status summaries

== Key admin endpoints

[cols="1,2,3", options="header"]
|===
| Method | Path | Purpose
| `GET /admin/system_status` | shell | Render operations dashboard.
| `POST /admin/system_status/test_connection` | action | Pre-flight-only SMTP handshake check (`SmtpConnectionValidator`); sends no email.
| `POST /admin/system_status/update_smtp` | action | Validate (`SmtpConnectionValidator`) and persist SMTP config via `SystemEmailConfig`.
| `POST /admin/system_status/test_email` | action | Send a real diagnostic email via `CentralNotificationMailer`.
| `GET /admin/reports` | shell | Analytics and export dashboard.
| `GET /admin/report_snapshots` | shell | Snapshot history.
| `GET /admin/intelligence` | shell | AI operations console.
| `GET /admin/migrations` | shell | Connectors, ingestion, and data-health console.
|===

NOTE: Most admin routes render HTML shells; the heavy-lifting JSON traffic goes
through `/api/v1/...` controllers documented in <<section-api-reference>>.
