= Recycle Bin

The recycle bin is built on the `SoftDeletable` concern and provides a delayed,
recoverable delete workflow for both assets and folders.

== Soft delete model

`app/models/concerns/soft_deletable.rb` adds the common lifecycle:

[source,ruby]
----
record.soft_delete
record.restore
record.trashed?
Model.active
Model.trashed
----

== Purge lifecycle

`BinPurgeService` performs the irreversible phase. It checks for workflow
protection, cleans collection/duplicate references, and records reclaimed
storage metrics before the hard delete is finalized.

[cols="1,3", options="header"]
|===
| Worker/service | Role
| `BinPurgeWorker` | Runs retention-based purge batches.
| `TrashCleanupWorker` | Cleans remaining stale trash artifacts.
| `BinPurgeService` | Domain rules and safety checks for permanent delete.
|===

== REST endpoints

[cols="1,2,3", options="header"]
|===
| Method | Path | Description
| `GET` | `/api/v1/bin` | List trashed assets/folders (search, `type` filter incl. `other`, sort, pagination).
| `GET` | `/api/v1/bin/stats` | Totals, storage used, retention days.
| `POST` | `/api/v1/bin/bulk_restore` | Restore mixed selections.
| `DELETE` | `/api/v1/bin/bulk_destroy` | Permanently delete selected rows.
| `DELETE` | `/api/v1/bin/empty` | Empty the entire bin.
| `GET/PATCH` | `/api/v1/bin/retention_policy` | Read/update purge policy.
| `POST` | `/api/v1/bin/trigger_purge` | Start a purge run.
| `GET` | `/api/v1/bin/purge_status` | Job lock/status payload.
|===

== Bin UI

`BinManager.jsx` is a full-screen manager with:

* stats bar and retention summary
* list/grid toggle
* mixed asset/folder selection
* bulk restore and bulk permanent delete
* active purge banner while a job is running

`BinFilterBar.jsx` mirrors the single-row filter/toolbar pattern used by the
Folders explorer (`AssetFilterBar.jsx`): search, a type-filter dropdown (All
Items / Assets / Folders / Images / Videos / Documents / Other), a sort
dropdown, a result count, a per-page selector (25 / 50 / 100 — passed through
to `GET /api/v1/bin`'s `per_page` param), an S/M/L grid-size toggle (grid view
only), and the grid/list view toggle — all kept on one scrollable row so the
page keeps more vertical space for content. Pagination itself uses MUI's
`Pagination` component (numbered pages + first/last buttons) driven by the
`{ page, pages }` values in the `/api/v1/bin` response.

The "Other" type filter (`type=other` on `GET /api/v1/bin`) is a catch-all
bucket for trashed assets whose content type is not an image, a video, or a
recognised document (e.g. audio files, archives, or assets with a missing/
unrecognised content type) — see `Api::V1::BinController#filter_by_media_type`.

The "View Trash Bin" button in the folder explorer's top bar
(`ExplorerTopBar.jsx`) navigates to the dedicated `/bin` page (full
navigation, `window.location.href = '/bin'`), which renders `BinManager.jsx`.
It previously toggled an in-explorer `viewMode = 'bin'` state that tried to
render `GET /api/v1/bin`'s `{ items, pagination }` payload through
`AssetExplorer.jsx`'s `{ folders, assets, breadcrumbs }`-shaped rendering
pipeline — an incompatible shape that always produced a blank grid. Navigating
to the real `/bin` page avoids maintaining two separate bin-rendering
implementations.

WARNING: Permanent delete is intentionally separated from the initial trash
operation; workflow-protected assets may remain unpurgeable until instances are
completed or cancelled.
