= Workflow Engine

The Capri DAM workflow engine lets administrators define **directed acyclic graph
(DAG)** approval and automation flows that attach to assets or collections.
Workflows are built in the browser using the **Workflow Designer** (`@xyflow/react`)
and executed by the `WorkflowEngineWorker` / `WorkflowInitiatorWorker` Sidekiq workers.

== Concepts

[cols="1,3", options="header"]
|===
| Term | Description

| **Workflow** | A named template containing a DAG of `WorkflowStep` records. Stored in the DB; created and edited in the Workflow Designer UI.
| **WorkflowInstance** | A live execution of a `Workflow` attached to an `Asset` or `Collection`. Transitions through step states.
| **WorkflowStep** | A node in the template DAG. Has a `step_type` and optional conditions.
| **WorkflowTask** | A human-action record created when an `approval` step is reached. Assigned to a user or group.
|===

== Step types

[cols="1,3", options="header"]
|===
| `step_type` | Behaviour

| `start` | Entry node; every workflow has exactly one.
| `approval` | Creates a `WorkflowTask` for the configured `assignee_group`. Instance pauses until task is resolved.
| `notification` | Sends an in-app + email notification via `TaskNotificationWorker`. Non-blocking.
| `auto_archive` | Automatically sets `asset.status = :archived`.
| `auto_publish` | Sets `asset.status = :active` and triggers CDN warm-up.
| `condition` | Evaluates a metadata predicate; branches to one of two outgoing edges.
| `end` | Terminal node; marks `WorkflowInstance` as `completed`.
|===

== Workflow lifecycle

[source]
----
Asset uploaded / event triggered
        │
        ▼
WorkflowInitiatorWorker.perform_async(asset_id, workflow_id)
  → creates WorkflowInstance (status: running)
  → creates first WorkflowTask if step_type = approval
        │
        ▼ (async, Sidekiq)
WorkflowEngineWorker.perform_async(instance_id)
  → evaluates current step
  → if approval: wait (task must be resolved via PATCH /api/v1/workflow_tasks/:id)
  → if auto-step: execute and advance to next step
  → if end: mark instance completed
----

== Trigger rules

A `Workflow` can have trigger conditions evaluated by `WorkflowEvaluatorService`:

[source,json]
----
{
  "trigger": "asset.created",
  "conditions": [
    { "field": "folder.path", "op": "starts_with", "value": "marketing/" },
    { "field": "content_type", "op": "in", "value": ["image/jpeg","image/png"] }
  ]
}
----

`AssetWorkflowTriggerWorker` runs after `AssetProcessorWorker` and checks all
active `Workflow` records against the new asset.

== Recycle Bin integration

When `bin_retention_policy.workflow_behavior = "block"`:

* Assets with active `WorkflowInstance` records cannot be permanently purged by `BinPurgeWorker`.
* They remain in the bin with `protected_by_workflow: true` until all instances complete or are cancelled.

== Workflow Designer (UI)

The `Workflows/WorkflowDesigner.jsx` canvas exposes:

* **Add node** — drag from the palette to the canvas.
* **Connect nodes** — draw edges between node handles.
* **Configure node** — click a node to open its property panel (assignee, conditions, notification text).
* **Save** — serialises the graph to JSON and calls `POST/PATCH /api/v1/workflows`.
* **Test run** — starts a `WorkflowInstance` against a selected test asset.

The designer shell is organised into three sections under a sticky action bar
(name, active toggle, live step-count badge, save button):

. **Blueprint Definition** — name + description.
. **Trigger & Scope** — trigger event and folder include/exclude scope.
. **Branching Logic Canvas** — the React Flow graph.

[NOTE]
====
Escalation is configured **per approval step** (on each `ApprovalNode`), not at
the workflow level.  The previous workflow-wide "System-Wide Escalation" panel
was removed in Designer v2; the `workflows.fallback_assignee_*` columns are
retained only for backward-compatibility and are no longer exposed in the UI or
permitted by `WorkflowsController#workflow_params`.  See
`WorkflowAdvancerService` for the resolution order:
**step assignee → step fallback → (legacy) workflow fallback**.
====

Node components:

[cols="1,3", options="header"]
|===
| Component | Description

| `nodes/StartNode.jsx` | Green start bubble; no configuration.
| `nodes/ApprovalNode.jsx` | Assignee (user/group), logic (any/all), SLA in days, and a step-level **fallback assignee** for escalation.
| `nodes/EndNode.jsx` | Red end bubble; shows completion action (archive / publish / none).
|===

== WorkflowTask resolution

An assigned user sees pending tasks on `GET /api/v1/workflow_tasks`.
To complete a task:

[source,http]
----
PATCH /api/v1/workflow_tasks/99
Content-Type: application/json

{ "decision": "approve", "comment": "LGTM" }
----

`WorkflowEngineWorker` is re-queued after resolution and advances the instance.

== Service objects

[cols="2,3", options="header"]
|===
| Service | Description

| `WorkflowEvaluatorService` (`app/services/workflow_evaluator_service.rb`) | Evaluates trigger conditions and step predicates.
| `WorkflowInitiatorWorker` | Creates `WorkflowInstance` and seeds initial tasks.
| `WorkflowEngineWorker` | State-machine runner; advances or completes instances.
| `TaskNotificationWorker` | Sends notifications to task assignees.
|===
