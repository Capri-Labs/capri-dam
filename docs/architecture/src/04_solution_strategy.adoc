== Solution Strategy

The Capri DAM follows a "Thin Origin, Fat Edge" strategy. By offloading heavy processing to the CDN and delegating binary storage to specialized cloud providers, the core system remains lightweight, scalable, and inexpensive to operate.

=== Core Architectural Decisions

[cols="1,2,2", options="header"]
|===
| Goal/Constraint | Strategic Decision | Rationale
| **Cost Efficiency** | **On-the-fly Transformation** | Instead of storing multiple renditions of an asset, the system generates them dynamically via Fastly IO. This reduces storage costs by 90% and eliminates "rendition bloat."
| **Performance** | **Edge-Based Rendering** | Moving image processing to the Fastly network edge ensures that transformations happen geographically close to the user, reducing latency.
| **Flexibility** | **Metadata-Driven URLs** | The system uses a centralized database to map business requirements (e.g., "High-quality banner") to technical URL parameters (e.g., `?width=1200&auto=webp`).
| **Cloud Agnosticism** | **S3-Compatible Abstraction** | By using a standardized storage interface, the system can switch between AWS, Azure, and GCP without refactoring the core logic.
|===

=== Key Technical Strategies

==== 1. Headless URL Generation
The system does not serve the image binary itself. Instead, the API acts as a "Signed URL Generator." It looks up an asset's focal point and metadata in the database and constructs a secured Fastly IO URL. This keeps the API's bandwidth usage near zero.

==== 2. Smart Cropping via Metadata
Rather than using proprietary AI engines, the system stores **focal point coordinates (X, Y)** in the metadata. These coordinates are passed to Fastly's `crop` parameter, ensuring that the "Smart Crop" logic is consistent across all device sizes without manual intervention.

==== 3. Search-First Discovery
To handle large asset libraries, we utilize a "Sidecar Search Index" (**Typesense**). Every time an asset is tagged or uploaded, the Core API asynchronously updates the search index, enabling instant discovery without taxing the relational database (PostgreSQL).

==== 4. Preventive Security (URL Signing)
To prevent "Image Bombing" attacks (where malicious users request thousands of different sizes to drive up costs), the system implements **HMAC URL Signing**. Fastly will only process and cache requests that carry a valid signature generated by the Capri DAM API.

==== 5. Cache Invalidation Strategy
The system utilizes **Surrogate Keys**. Each asset is tagged with a unique ID at the Fastly edge. When a user updates an asset in the GUI, the system sends a single "Purge by Key" request to Fastly, instantly clearing all renditions of that asset globally.
