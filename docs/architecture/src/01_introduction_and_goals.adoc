== Introduction and Goals

=== Executive Summary

The **Capri DAM** project is a high-performance, cost-optimised Digital Asset
Management system built on **Ruby on Rails 8.1** (backend) and **React 19** with
Material UI v9 (frontend), deployed on **Ruby 4.0.3**.

It modernises media workflows by providing advanced dynamic media features — real-time
image transformation, smart cropping, structured metadata, role-based access control
(RBAC), and global delivery — without the high entry costs and rigid licensing
associated with legacy monolithic media suites.

By leveraging **Fastly Image Optimizer (IO)** and a "Bring Your Own Storage" (BYOS)
model, the system enables organisations to maintain full ownership of their master
assets while delivering perfectly optimised media via a metadata-driven, edge-first
architecture.

=== Requirements Overview

The primary goal is to manage the lifecycle of a digital asset from upload to global
delivery with minimal latency.

* **Multi-Cloud Storage:** Support for AWS S3, Azure Blob Storage, and Google Cloud Storage.
* **Metadata Management:** Store and index technical (dimensions, EXIF/IPTC) and business (tags, focal points, schemas) metadata.
* **Dynamic Transformation:** Real-time cropping, resizing, format conversion, and WebP delivery at the Edge via Fastly IO.
* **Rich GUI:** Material UI-based interface for non-technical users to browse folders, upload assets, and configure image/video profiles.
* **High-Speed Search:** Faceted lexical search and metadata filtering directly
  against PostgreSQL, plus pgvector-backed semantic/visual similarity search
  (cosine distance over CLIP-style embeddings) — no external search service.
* **Cache Management:** Instant per-asset Fastly purge via Surrogate Keys.
* **RBAC Access Control:** Hierarchical user groups with per-folder permission policies (7-column ACL matrix).
* **Metadata Schemas:** Configurable field schemas per folder path for structured metadata.
* **Image Profiles:** Named Fastly IO transformation presets (width, crop, format).
* **Video Profiles:** Encoding presets with folder-path assignment for transcoding workflows.

=== Quality Goals

[cols="1,2,2", options="header"]
|===
| Priority | Quality Goal | Description

| 1 | **Cost Efficiency**
| Minimise fixed licensing costs; favour a "pay-as-you-go" operational model.
  On-the-fly Fastly IO rendering eliminates ~90% of rendition storage costs.

| 2 | **Performance**
| Image transformations execute at the Edge in <100 ms for non-cached requests.
  Background workers use queue weighting to prevent bulk uploads from blocking
  urgent user-facing jobs.

| 3 | **Security**
| HMAC-signed Fastly URLs prevent denial-of-wallet attacks.
  RBAC with explicit-deny overrides ensures principle of least privilege.
  All endpoints secured via Devise + optional Keycloak SSO (OmniAuth).

| 4 | **Flexibility**
| Headless architecture — any frontend (web, mobile, IoT) can consume assets
  via REST or GraphQL APIs. Storage provider is swappable without code changes.

| 5 | **Scalability**
| Sidekiq queue topology absorbs traffic spikes; Fastly CDN shields the origin
  from delivery load. Architecture supports horizontal scaling via Kamal.
|===
