== Deployment View

The Capri DAM ships as a single Docker image, deployed with **Kamal 2**
(`config/deploy.yml`) to one or more plain Linux servers (VPS or bare metal â€”
no managed PaaS/serverless required). `kamal-proxy` handles TLS termination
(automatic Let's Encrypt), zero-downtime rollouts, and load balancing. See
link:../deployment-guide-kamal/[the Kamal Deployment Guide] for the full
operational playbook (prerequisites, secrets, first deploy, routine
deploys/rollback, SSL, monitoring, backups). Two managed-infrastructure
alternatives are also documented: link:../deployment-guide-aws/[AWS (ECS
Fargate)] and link:../deployment-guide-azure-kubernetes/[Azure Kubernetes
Service (AKS)].

=== Infrastructure Level 1: Reference Single-Host Topology

The starter `config/deploy.yml` pins the `web` and `worker` roles to the
**same host**, because the default Active Storage backend is local disk (a
bind-mounted volume both roles need to share). Switching the active
`StorageBackend` to S3/GCS/Azure Blob (no code changes â€” just an admin UI
setting) removes that constraint and lets `web`/`worker` scale onto separate
hosts.

[plantuml, target="deployment-level-1", format="svg"]
....
@startuml
skinparam componentStyle uml2

    [Web/Mobile Browser] as Browser
}

node "CDN Edge (Fastly / Cloudflare / Akamai)" {

        [TLS termination\n+ zero-downtime routing] as Proxy
    }
    package "App Containers" {
        [web: Puma + Thruster\n(Rails app + React SPA views)] as Web
        [worker: Sidekiq\n(no HTTP, proxy: false)] as Worker

        database "db: pgvector/pgvector:pg16" as DB
        database "redis: redis:7-alpine" as Redis

    [S3 / GCS / Azure Blob] as CloudStorage

Worker   --> DB
Worker   --> Redis
Web      --> LocalDisk : Read/write master\n(local StorageAdapter)
Web      --> CloudStorage : Read/write master\n(S3/GCS/Azure StorageAdapter)

=== Deployment Elements

[cols="1,2,2", options="header"]
|===
| Node / Artifact | Category | Description
| **kamal-proxy** | Reverse proxy | Runs on each Kamal host; terminates TLS via automatic Let's Encrypt, load-balances across app replicas, and performs zero-downtime cutover on deploy (health-checked against `GET /up`).
| **web role** | App container | Puma (`WEB_CONCURRENCY=2`, `RAILS_MAX_THREADS=5`) behind Thruster (`bin/thrust`, listens on container port 80). Serves the Rails-rendered HTML shell + mounted React islands, REST API v1, and GraphQL.
| **worker role** | App container | `bundle exec sidekiq -C config/sidekiq.yml`. No HTTP server â€” `proxy: false` in `config/deploy.yml` so kamal-proxy never routes to it. Pinned to the same host as `web` by default (shared local-disk volume); can move to its own host once storage is cloud-backed.
| **db accessory** | Kamal accessory (Docker) | `pgvector/pgvector:pg16` image â€” PostgreSQL 16 with the pgvector extension preinstalled, so semantic/visual similarity search (see <<section-design-decisions>>, ADR 3) needs no extra setup. Bound to `127.0.0.1:5432` on the host (admin `psql` access only); the app reaches it via the private Docker network alias `capri-dam-db`.
| **redis accessory** | Kamal accessory (Docker) | `redis:7-alpine`. Backs Sidekiq queues, Action Cable, and Coverband (separate DB index) via `REDIS_URL`/`COVERBAND_REDIS_URL`.
| **Bind-mounted volume** | Host storage | `/var/lib/capri-dam/storage` on the host â†” `/rails/storage` in every container. Used by the local `StorageAdapter`; irrelevant once a cloud `StorageBackend` (S3/GCS/Azure) is made active.
| **Object Storage (optional)** | Cloud storage | S3 / GCS / Azure Blob, selected via the admin-configurable `StorageBackend` record â€” no code change or redeploy needed to switch.
| **CDN Edge** | Edge Computing | Fastly (default), Cloudflare, or Akamai â€” selected via the pluggable `CdnAdapters::*` (see <<section-building-block-view>>). Handles image/video transformation and caching; falls back to a no-op `NullAdapter` when no CDN is configured (e.g. local dev).
| **Container Registry** | GHCR | `.github/workflows/docker.yml` builds and pushes multi-arch images to `ghcr.io/capri-labs/capri-dam` on every push to `main`/`master` (tag `latest`) and on semver tags (`v*.*.*`). Kamal pulls from there by default (or builds locally with the same `RUBY_VERSION`/`NODE_VERSION` build args).
|===

=== Scaling Path

The starter configuration favours "one small VPS, minimal moving parts" over
premature horizontal scaling â€” consistent with the <<section-constraints,budget
and small-team constraints>> â€” but each step below is a config change, not a
rewrite:

1. **Vertical first:** bump `WEB_CONCURRENCY`/`RAILS_MAX_THREADS` and the
   server's CPU/RAM tier before adding hosts.
2. **Split worker from web:** once storage is cloud-backed (S3/GCS/Azure â€”
   already supported), move the `worker` role's `hosts:` to a second server
   so heavy Sidekiq jobs (bulk downloads, ingestion, video transcode) don't
   compete with Puma for CPU.
3. **Scale web horizontally:** add more `web` hosts under `servers.web.hosts`
   â€” `kamal-proxy` load-balances automatically; Postgres/Redis accessories
   stay on their own host(s) as the app scales out.
4. **Externalise Postgres/Redis:** swap the `db`/`redis` Kamal accessories
   for a managed service (RDS/Cloud SQL + pgvector-enabled instance,
   ElastiCache/Memorystore) once self-managed accessory operations
   (backups, failover) become the bottleneck â€” no application code changes
   required either way.
