<table class="compare-table">
  <caption>🧭 Choosing a deployment target</caption>
  <thead>
    <tr>
      <th>Criteria</th>
      <th>Kamal (VPS/bare metal)</th>
      <th>AWS (ECS Fargate)</th>
      <th>Azure (AKS)</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Ops complexity</td>
      <td>Lowest — one/few servers, no orchestrator to learn</td>
      <td>Medium — no servers to patch, but ECS/ALB/IAM concepts to learn</td>
      <td>Highest — full Kubernetes (manifests, Helm, RBAC) to operate</td>
    </tr>
    <tr>
      <td>Team skill fit</td>
      <td>Best if the team is comfortable with plain Linux servers &amp; Docker</td>
      <td>Best if the team is AWS-native but doesn't want to run Kubernetes</td>
      <td>Best if the team already runs/knows Kubernetes elsewhere (portability matters)</td>
    </tr>
    <tr>
      <td>Scaling model</td>
      <td>Manual — add servers/roles as needed</td>
      <td>Automatic — ECS service desired-count + Fargate capacity</td>
      <td>Automatic — <code>HorizontalPodAutoscaler</code> + cluster autoscaler</td>
    </tr>
    <tr>
      <td>Managed Postgres/Redis</td>
      <td>No — self-hosted as Kamal accessories (you manage backups/patching)</td>
      <td>Yes — Aurora PostgreSQL + ElastiCache (pgvector-compatible)</td>
      <td>Yes — Flexible Server + Azure Cache for Redis (pgvector-compatible)</td>
    </tr>
    <tr>
      <td>Object storage</td>
      <td>Local disk by default (bind mount), or switch to S3/Blob in-app</td>
      <td>S3 (in-app <code>StorageBackend</code> switch, no code change)</td>
      <td>Azure Blob (in-app <code>StorageBackend</code> switch, no code change)</td>
    </tr>
    <tr>
      <td>Relative baseline cost</td>
      <td>Lowest fixed cost (one small/medium VPS)</td>
      <td>Medium (Fargate vCPU-seconds + NAT Gateway + Aurora/ElastiCache)</td>
      <td>Medium–higher (AKS node pool + NAT + Flexible Server + Azure Cache)</td>
    </tr>
    <tr>
      <td>Deploy/rollback mechanism</td>
      <td><code>bin/kamal deploy</code> / <code>bin/kamal rollback</code></td>
      <td><code>register-task-definition</code> + <code>update-service</code> / prior revision ARN</td>
      <td><code>kubectl set image</code> / <code>kubectl rollout undo</code></td>
    </tr>
    <tr>
      <td>Portability if you leave the cloud</td>
      <td>N/A — already provider-agnostic (any VPS)</td>
      <td>Low — ECS/Fargate resources are AWS-specific</td>
      <td>High — the same Kubernetes manifests run on EKS/GKE/any managed K8s with minimal changes</td>
    </tr>
    <tr>
      <td>Full guide</td>
      <td><a href="deployment-guide-kamal/">Kamal Deployment Guide</a></td>
      <td><a href="deployment-guide-aws/">AWS Deployment Guide</a></td>
      <td><a href="deployment-guide-azure-kubernetes/">Azure/Kubernetes Deployment Guide</a></td>
    </tr>
  </tbody>
</table>

  infrastructure differs, and switching between them later does not require any application code
  changes.
</p>
