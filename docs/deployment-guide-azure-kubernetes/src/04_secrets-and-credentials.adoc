[[section-azure-k8s-secrets-and-credentials]]
== Secrets & Credentials

=== `RAILS_MASTER_KEY`

Capri DAM decrypts `config/credentials.yml.enc` using `RAILS_MASTER_KEY` at boot — this is the one
secret that must never be baked into the image or committed. Two supported approaches:

==== Option A — Azure Key Vault + Secrets Store CSI Driver (recommended for production)

[source,bash]
----
az keyvault create --resource-group capri-dam-rg --name capri-dam-kv --location eastus
az keyvault secret set --vault-name capri-dam-kv --name rails-master-key \
  --value "$(cat config/master.key)"
az keyvault secret set --vault-name capri-dam-kv --name database-url --value "$DATABASE_URL"
az keyvault secret set --vault-name capri-dam-kv --name redis-url --value "$REDIS_URL"

az aks addon enable --resource-group capri-dam-rg --name capri-dam-aks \
  --addon azure-keyvault-secrets-provider

az aks update --resource-group capri-dam-rg --name capri-dam-aks --enable-workload-identity
----

Then mount a `SecretProviderClass` that syncs the Key Vault secrets into a Kubernetes `Secret`
(`capri-dam-secrets`) referenced by the `Deployment` manifests in
<<section-azure-k8s-cluster-and-manifests>>:

[source,yaml]
----
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata: { name: capri-dam-kv, namespace: capri-dam }
spec:
  provider: azure
  secretObjects:
    - secretName: capri-dam-secrets
      type: Opaque
      data:
        - objectName: rails-master-key
          key: rails-master-key
        - objectName: database-url
          key: database-url
        - objectName: redis-url
          key: redis-url
  parameters:
    usePodIdentity: "false"
    useVMManagedIdentity: "true"
    userAssignedIdentityID: "$AKS_KUBELET_IDENTITY_CLIENT_ID"
    keyvaultName: capri-dam-kv
    tenantId: "$AZURE_TENANT_ID"
    objects: |
      array:
        - |
          objectName: rails-master-key
          objectType: secret
        - |
          objectName: database-url
          objectType: secret
        - |
          objectName: redis-url
          objectType: secret
----

==== Option B — plain Kubernetes `Secret` (simpler, fine for staging)

[source,bash]
----
kubectl create secret generic capri-dam-secrets --namespace capri-dam \
  --from-literal=rails-master-key="$(cat config/master.key)" \
  --from-literal=database-url="$DATABASE_URL" \
  --from-literal=redis-url="$REDIS_URL"
----

Kubernetes `Secret` objects are only base64-encoded, not encrypted, unless the cluster has
**encryption at rest for etcd** enabled (AKS enables this by default) — Option A is still preferable
for production since it also gives you Key Vault's audit log and rotation support.

=== `ConfigMap` for non-sensitive configuration

[source,bash]
----
kubectl create configmap capri-dam-config --namespace capri-dam \
  --from-literal=RAILS_ENV=production \
  --from-literal=RAILS_LOG_TO_STDOUT=true \
  --from-literal=RAILS_SERVE_STATIC_FILES=true
----

=== Workload identity for Blob Storage access

Rather than embedding a Storage Account connection string, prefer **Azure AD Workload Identity**: bind
a federated identity credential to the AKS service account used by the `web`/`worker` pods, and grant
that managed identity the `Storage Blob Data Contributor` role scoped to the specific storage account
from <<section-azure-k8s-managed-data-services>> — nothing broader.

[source,bash]
----
az identity create --resource-group capri-dam-rg --name capri-dam-workload-identity
az role assignment create --assignee "$WORKLOAD_IDENTITY_PRINCIPAL_ID" \
  --role "Storage Blob Data Contributor" \
  --scope "$STORAGE_ACCOUNT_ID"
----
